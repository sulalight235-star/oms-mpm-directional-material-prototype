# OMS-MPM 定向采购物料同步及自动化审核需求文档

> **系统链路说明**：本需求中，OMS（订单系统）通过 **H0 平台**与 MPM（物料主数据系统）进行对接。OMS 不直接调用 MPM 接口，所有物料数据的推送与编码回传均经由 H0 平台中转。文档中"OMS → MPM"的表述均指该链路，不代表直连。

## 1. 业务痛点分析
- **查重颗粒度不足**：传统匹配仅靠名称，易造成“一物多码”，增加库存管理难度。
- **进度不可控**：销售发起申请后无法实时感知 MPM 审核进度，缺乏有效的在线催办机制。
- **数据不一致**：跨系统传输中，单位、品牌等基础维度不统一，导致物料主数据治理混乱。

------

## 2. 需求概述

本需求旨在通过接口自动化手段，实现 OMS 定向函物料经由 **H0 平台**在 MPM 中的**预判重、AI 补全、批量审核与整单回传**。核心原则为：**源头库对齐、七要素严判重、批次强一致性。**

**实现 OMS 定向函物料经 H0 平台在 MPM 系统中的自动化标准化流程。通过”查重优先+AI 辅助+人工校验”的组合拳，确保物料编码快速、准确地经 H0 回流至 OMS，支撑销售业务**

##  OMS 与 MPM 系统交互全局流程（经由 H0 平台）

### 1. 源头约束（OMS 发起端）

- **标准化录入**：OMS 在填写定向函时，通过 H0 平台调用 MPM 的基础字典。
  - **品牌**：必须从 MPM 品牌库中选择。
  - **单位**：必须从 MPM 单位库中选择。
- **交互控制**：OMS 端在点击”提交定向函”后，**立即置灰**提交按钮，进入”MPM 审核中”状态，防止重复触发。

### 2. MPM 接收与初步处理

- **接口入参**：MPM 通过 H0 平台接收 OMS 推送的 `Batch_ID`、`Material_ID` 及标准化后的物料 7 要素。
- **幂等控制**：MPM 接收时校验 `Batch_ID`。若该批次已在流程中（审核中/已完成），直接拒绝请求并返回状态码，确保流程锁死。

### 3. 数据匹配与分流

MPM 获取数据后，不是立即执行 7 要素查重，而是先完成**类目智能划分**，再进入匹配。

- **第一步：类目智能划分**
  - AI 先预测推荐四级类目及匹配度。
  - **匹配度 ≥ 90%**：系统自动采用推荐四级类目。
  - **匹配度 < 90%**：必须由采购人员人工确认四级类目，确认完成后才能进入下一步。

- **第二步：7 要素匹配**
  - 在四级类目已确认的前提下，系统按固定 7 要素执行匹配：
    `【品牌 + 四级类目 + 名称 + 规格 + 型号 + 单位 + 描述】`。
  - **冲突降级**：若命中多个（概率极低），取**匹配度最高**（或创建时间最新）的一个。

- **第三步：逻辑分流**
  - **7 要素命中**：复用已有咸亨物料编码。
  - **7 要素未命中**：进入【OMS 定向物料（新增审核）】列表，由采购人员人工审核通过后新增编码。

##  2. MPM 内部执行业务流程逻辑

MPM 内部通过“四步走”战略，实现类目前置确认、存量复用与增量审核。

### 第一步：类目智能划分与确认 (Tier 1)

- **AI 预判**：系统先根据名称、型号、描述等信息预测推荐四级类目，并输出匹配度。
- **自动采用规则**：若匹配度 **≥ 90%**，系统自动采用推荐四级类目。
- **人工确认规则**：若匹配度 **< 90%**，必须由采购人员人工确认四级类目；四级类目未确认前，不得进入后续 7 要素匹配。

### 第二步：基于已确认类目的 7 要素匹配 (Tier 2)

- **逻辑**：系统以 `【品牌 + 四级类目 + 名称 + 规格 + 型号 + 单位 + 描述】` 为唯一性键值进行全局检索。
- **冲突处理**：若命中 ≥ 2 个编码，系统自动关联**匹配度最高**（或最新使用）的一个。
- **状态标记**：命中编码的物料标记为“系统自动匹配”；未命中的进入下一环节。

### 第三步：人工审核与编码新增 (Tier 3)

- **聚合审核**：审核员以 `Batch_ID` 为维度进入详情页。
- **交互修正**：审核员补全缺失信息，并对未命中的物料进行审核确认。
- **终态赋码**：仅对 7 要素未命中的物料，在审核通过后生成新的咸亨物料编码；若有任何必填项为空，系统**强行阻断**提交并高亮报错。

### 第四步：编码写入、全单检验与回传 H0 (Tier 4)

- **编码写入**：每一行物料获得咸亨物料编码后，立即将 `定向函行号（物料 ID） → 咸亨物料编码` 写入关系表；若该行已有旧值，**直接覆盖**，不做冲突拦截。
- **全单检验**：每次写入后触发检验：检查当前 `Batch_ID` 下所有物料行是否均已存在咸亨物料编码。
  - **全部存在**：立即触发整单回传，将全量咸亨物料编码一次性回传至 H0（OMS 定向函系统）。
  - **仍有缺失**：等待，不执行回传。
- **触发时机**：该检验在以下两个节点均会触发：
  1. 7 要素匹配命中、复用已有编码之后
  2. 采购审核通过、生成新编码之后

## 核心逻辑规则清单 (Core Logic Rules)

| **维度**       | **规则名称**         | **详细说明**                                                                 |
| -------------- | -------------------- | ---------------------------------------------------------------------------- |
| **唯一性校验** | **七要素严匹配**     | 判定依据固定为：`品牌 + 四级类目 + 名称 + 规格 + 型号 + 单位 + 描述`。七项完全一致则视为存量复用。 |
| **类目前置**   | **类目确认先行**     | 四级类目必须先确认后，才能进入 7 要素匹配；类目未确认时不得继续流转。         |
| **阈值分流**   | **90% 自动采用规则** | AI 推荐四级类目匹配度 **≥90%** 自动采用；**<90%** 必须由采购人员人工确认。      |
| **冲突处理**   | **择优关联逻辑**     | 若 7 要素匹配命中了 2 个以上编码，系统自动关联**匹配度最高**的一个；审核员在界面拥有最终切换权。 |
| **编码生成**   | **未命中后人工新增** | 7 要素未匹配到现有编码时，必须由采购人员人工审核通过后，系统才允许新增咸亨物料编码。 |
| **编码写入**   | **覆盖写入规则**     | `定向函行号 → 咸亨物料编码` 写入关系表时，直接覆盖已有旧值，不做冲突拦截。主数据层编码本身允许复用。 |
| **全单检验**   | **编码完整性检验**   | 每次行级编码写入后，系统自动检验当前 `Batch_ID` 下所有物料行是否均已存在咸亨物料编码；全部存在则立即触发整单回传至 H0，否则继续等待。 |
| **回传触发**   | **双节点触发机制**   | 以下两个节点均会触发全单检验：① 7 要素命中复用编码后；② 采购审核通过新增编码后。 |
| **事务控制**   | **Batch 强一致性**   | 整单回传以 `Batch_ID` 为单位，仅在全单编码完整时一次性回传全量咸亨物料编码至 H0，确保 OMS 定向函状态同步更新。 |
| **并发保护**   | **流程幂等锁**       | 以 `Batch_ID` 为主键。MPM 接收请求后立即锁定该单号，流程结束前拒绝任何重复推送。 |
| **时效响应**   | **钉钉待办推送**     | OMS 经由 H0 平台触发催办，推送包含 `Batch_ID` 跳转链接的待办给审核组，实现任务闭环。 |

## 异常场景处理 (Exception Handling)

1. **类目匹配度不足 90%**：不得自动进入 7 要素匹配，必须停留在采购人员人工确认四级类目节点。
2. **必填项缺失**：即使 OMS 校验漏过，MPM 审核点击”通过”时会再次进行**强校验**，缺值则报错并定位单元格。
3. **回传失败重试**：MPM 记录回传日志，若回传 H0 失败，系统自动重试 3 次，并在审核列表提供”手动补推”按钮。
4. **重复提交**：若销售在置灰前快速双击，MPM 侧通过 `Batch_ID` 唯一键冲突直接拦截。



##接口清单

Custom Gem

基于 PRD 中的业务逻辑，MPM 需要通过 **H0 平台**对外暴露**两个核心接口**：一个用于接收 OMS 经 H0 推送的物料数据，一个用于接收钉钉催办请求。同时，MPM 处理完成后需**经 H0 回调** OMS 的一个接口。

以下是具体的接口定义建议（采用标准 JSON 格式）：

------

## 1. 【接收】OMS 定向物料推送接口（经 H0 平台）

**接口描述**：OMS 发起定向函时，经由 H0 平台将物料包推送到 MPM。MPM 先执行幂等检查，再进行类目智能划分与后续匹配流程。

- **请求方式**：`POST`
- **接口路径**：`/api/mpm/external/v1/directional/material/sync`

### 请求报文 (Request Body)

JSON

```
{
  "batchId": "DX202603310001",           // 定向函单号 (Batch_ID)
  "applicantId": "U9527",                // 发起人人员ID
  "applicantName": "张三",                // 发起人姓名
  "submitTime": "2026-03-31 14:00:00",   // 提交时间 (YYYY-MM-DD HH:mm:ss)
  "sourceSystem": "OMS",
  "materials": [
    {
      "omsMaterialId": "LINE_001",       // OMS行唯一标识
      "mpmMaterialCode": "A1901005067",  // MPM物料编码（咸亨物料编码）
      "itemName": "气垫",                 // 物料名称
      "model": "NX20663016",             // 规格型号
      "brandId": "B7788",                // 品牌ID (MPM库对齐)
      "unitId": "U001",                  // 单位ID (MPM库对齐)
      "quantity": "100",                 // 数量 (String)
      "purchasePrice": "100.00",         // 进货单价 (String)
      "description": "云游充气床，蓝"       // 描述
    }
  ]
}
```

### 响应报文 (Response)

- **Success**: `{"code": 200, "message": "接收成功，正在处理", "data": {"status": "PROCESSING"}}`
- **Error (重复提交)**: `{"code": 400, "message": "该单号已在处理中，请勿重复提交"}`



## 2. 【接收】OMS 催办指令接口（经 H0 平台）

**接口描述**：销售在 OMS 点击催办，经由 H0 平台调用此接口告知 MPM 触发钉钉推送。

- **请求方式**：`POST`
- **接口路径** : `/api/mpm/external/v1/directional/material/urge`

### 请求报文 (Request Body)

JSON

```
{
  "batchId": "DX202603310001",
  "urgeUserId": "U9527",
  "urgeUserName": "张三"
}
```

### 响应报文 (Response)

- **Success**: `{"code": 200, "message": "催办消息已发送至审核员钉钉"}`

- **催办受限 (429 Too Many Requests)**: `{"code": 429, "message": "催办过于频繁，请在30分钟后再试"}`
