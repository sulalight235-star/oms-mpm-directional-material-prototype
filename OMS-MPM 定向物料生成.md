# OMS-MPM 定向采购物料同步及自动化审核需求文档

## 1. 业务痛点分析

- **数据孤岛与重复录入**：销售在 OMS 下单需物料编码，以往需手动在两个系统间切换录入，效率低下且易出错。
- **查重颗粒度不足**：传统匹配仅靠名称，易造成“一物多码”，增加库存管理难度。
- **进度不可控**：销售发起申请后无法实时感知 MPM 审核进度，缺乏有效的在线催办机制。
- **数据不一致**：跨系统传输中，单位、品牌等基础维度不统一，导致物料主数据治理混乱。

------

## 2. 需求概述

本需求旨在通过接口自动化手段，实现 OMS 定向函物料在 MPM 中的**预判重、AI 补全、批量审核与整单回传**。核心原则为：**源头库对齐、七要素严判重、批次强一致性。**

**实现 OMS 定向函物料在 MPM 系统中的自动化标准化流程。通过“查重优先+AI 辅助+人工校验”的组合拳，确保物料编码快速、准确地回流至 OMS，支撑销售业务**

##  OMS 与 MPM 系统交互全局流程

### 1. 源头约束（OMS 发起端）

- **标准化录入**：OMS 在填写定向函时，通过接口调用 MPM 的基础字典。
  - **品牌**：必须从 MPM 品牌库中选择。
  - **单位**：必须从 MPM 单位库中选择。
- **交互控制**：OMS 端在点击“提交定向函”后，**立即置灰**提交按钮，进入“MPM 审核中”状态，防止重复触发。

### 2. MPM 接收与初步处理

- **接口入参**：MPM 提供专门的 `OMS定向物料新增接口`，接收包含 `Batch_ID`、`Material_ID` 及标准化后的物料 7 要素。
- **幂等控制**：MPM 接收时校验 `Batch_ID`。若该批次已在流程中（审核中/已完成），直接拒绝请求并返回状态码，确保流程锁死。

### 3. 数据匹配与分流

MPM 获取数据后，立即执行**七要素查重**。

- **查重维度**：`【品牌 + 类目 + 名称 + 规格 + 型号 + 单位 + 描述】`。
- **冲突降级**：若命中多个（概率极低），取**匹配度最高**（或创建时间最新）的一个。
- **逻辑分流**：
  - **全量命中**：若 Batch 内所有物料均命中，则不入审核池。
  - **含新物料**：任一物料未命中，则整单进入【OMS 定向物料（新增审核）】列表。

##  2. MPM 内部执行业务流程逻辑

MPM 内部通过“三步走”战略，实现存量复用与增量审核。

### 第一步：七要素静默查重 (Tier 1)

- **逻辑**：系统以 `【品牌+类目+名称+规格+型号+单位+描述】` 为唯一性键值进行全局检索。
- **冲突处理**：若命中 ≥2 个编码，系统自动关联**匹配度最高**（或最新使用）的一个。
- **状态标记**：命中编码的物料标记为“系统自动匹配”，未命中的进入下一环节。

### 第二步：AI 增强与预填充 (Tier 2)

- **AI 介入**：针对未命中物料，AI 根据名称/型号预测“四级类目”（强制取 Top 1）。
- **默认值补全**：系统自动填充货主（1025）、市场价/商城价（0）、起订量/毛重（1）、是否含运费/检测费等 10+ 项非核心必填字段。

### 第三步：人工批量审核 (Tier 3)

- **聚合审核**：审核员以 `Batch_ID` 为维度进入详情页。
- **交互修正**：审核员核对 AI 类目，补全因 OMS 传值异常导致的空缺项。
- **终态赋码**：审核员点击“通过”，系统对新物料自动申请编码。此时，若有任何必填项为空，系统**强行阻断**提交并高亮报错。

## 📝 3. “整单催办”钉钉待办设计建议

为了让审核员更高效，钉钉消息建议包含以下要素：

- **标题**：`【紧急催办】OMS定向采购单待审核 - [Batch_ID]`
- **正文**：
  - **发起人**：姓名
  - **发布内容**：您有一个定向函单据的物料待审核处理【定向函单号编码】，为了避免耽误业务进度，请尽快处理定向函物料审核处理；
  - **申请时间**：YYYY-MM-DD HH:mm
- **跳转链接**：点击直达 MPM 系统该 `Batch_ID` 的审核详情页。

## 核心逻辑规则清单 (Core Logic Rules)

| **维度**       | **规则名称**       | **详细说明**                                                 |
| -------------- | ------------------ | ------------------------------------------------------------ |
| **唯一性校验** | **七要素严查重**   | 判定依据：`品牌 + 类目 + 名称 + 规格 + 型号 + 单位 + 描述`。七项完全一致则视为存量复用。 |
| **冲突处理**   | **择优关联逻辑**   | 若查重命中了 2 个以上编码，系统自动关联**匹配度最高**的一个。审核员在界面拥有最终切换权。 |
| **事务控制**   | **Batch 强一致性** | **All-or-Nothing**：批次内只要有一个物料未审核通过，整单不回传。确保 OMS 定向函状态同步更新。 |
| **并发保护**   | **流程幂等锁**     | 以 `Batch_ID` 为主键。MPM 接收请求后立即锁定该单号，流程结束前拒绝任何重复推送。 |
| **智能填充**   | **AI 辅助补全**    | 针对新物料，AI 自动映射四级类目；审核员在 MPM 端的修正信息（如更正了型号）需一并回传 OMS。 |
| **时效响应**   | **钉钉待办推送**   | OMS 催办按钮调用钉钉接口，推送包含 `Batch_ID` 跳转链接的待办给审核组，实现任务闭环。 |

## 异常场景处理 (Exception Handling)

1. **必填项缺失**：即使 OMS 校验漏过，MPM 审核点击“通过”时会再次进行**强校验**，缺值则报错并定位单元格。
2. **回传失败重试**：MPM 记录回传日志，若回传 OMS 失败，系统自动重试 3 次，并在审核列表提供“手动补推”按钮。
3. **重复提交**：若销售在置灰前快速双击，MPM 侧通过 `Batch_ID` 唯一键冲突直接拦截。



##接口清单

Custom Gem

基于 PRD 中的业务逻辑，MPM 需要对外暴露**两个核心接口**：一个用于接收 OMS 推送的物料数据，一个用于接收钉钉催办请求。同时，MPM 需要在处理完成后**回调** OMS 的一个接口。

以下是具体的接口定义建议（采用标准 JSON 格式）：

------

## 1. 【接收】OMS 定向物料推送接口

**接口描述**：OMS 发起定向函时，将物料包推送到 MPM。MPM 执行幂等检查和自动查重。

- **请求方式**：`POST`
- **接口路径**：`/api/mpm/external/v1/directional/material/sync`

### 请求报文 (Request Body)

JSON

```
{
  "batchId": "DX202603310001",           // 定向函单号 (Batch_ID)
  "applicantId": "U9527",                // 发起人人员ID
  "applicantName": "张三",                // 发起人姓名
  "submitTime": "2026-03-31 14:00:00",   // 提交时间 (YYYY-MM-DD HH:mm:ss)
  "sourceSystem": "OMS",
  "materials": [
    {
      "omsMaterialId": "LINE_001",       // OMS行唯一标识
      "itemName": "气垫",                 // 物料名称
      "model": "NX20663016",             // 规格型号
      "brandId": "B7788",                // 品牌ID (MPM库对齐)
      "unitId": "U001",                  // 单位ID (MPM库对齐)
      "quantity": "100",                 // 数量 (String)
      "purchasePrice": "100.00",         // 进货单价 (String)
      "description": "云游充气床，蓝"       // 描述
    }
  ]
}
```

### 响应报文 (Response)

- **Success**: `{"code": 200, "message": "接收成功，正在处理", "data": {"status": "PROCESSING"}}`
- **Error (重复提交)**: `{"code": 400, "message": "该单号已在处理中，请勿重复提交"}`



## 2. 【接收】OMS 催办指令接口

**接口描述**：销售在 OMS 点击催办，调用此接口告知 MPM 触发钉钉推送。

- **请求方式**：`POST`
- **接口路径** : `/api/mpm/external/v1/directional/material/urge`

### 请求报文 (Request Body)

JSON

```
{
  "batchId": "DX202603310001",
  "urgeUserId": "U9527",
  "urgeUserName": "张三"
}
```

### 响应报文 (Response)

- **Success**: `{"code": 200, "message": "催办消息已发送至审核员钉钉"}`

- **催办受限 (429 Too Many Requests)**: `{"code": 429, "message": "催办过于频繁，请在30分钟后再试"}`

------
